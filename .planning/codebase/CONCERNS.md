# Codebase Concerns

**Analysis Date:** 2026-04-02

## Tech Debt

**Duplicate Script Files (Stepper vs. BLDC variants):**
- Issue: Multiple control design scripts exist in parallel directories with conflicting configurations
  - `scripts/RRpendulum_FSFB_controldesign_torque.m` loads BLDC params but exists in root scripts
  - `scripts/BLDC/RRpendulum_FSFB_controldesign_torque.m` is the BLDC variant
  - Similar duplication for `RRpendulum_UKF_design.m`, `RRpendulum_swingup_controldesign.m`, `RRpendulum_decoupledLuenberger.m`, and parameter files
- Files: `scripts/RRpendulum_*.m`, `scripts/BLDC/RRpendulum_*.m` (15+ duplicate pairs)
- Impact: Confusion about which script is authoritative; maintenance burden when parameters change; risk of running wrong controller design for hardware platform
- Fix approach: Consolidate into single scripts that load platform-specific params, or clearly rename and document root-level scripts as "deprecated stepper variants"

**Model Version Lock-in (R2025b only):**
- Issue: All `.slx` files are binary MATLAB Simulink models locked to R2025b
  - `CLAUDE.md` specifies "MATLAB R2025b or later"
  - Older `.r2024b` backup versions exist in `models/_Oldversions/`
  - No forward compatibility or ASCII alternative representation
- Files: `models/*.slx`, `models/BLDC/*.slx`, `resources/lib/digtwin_labo_lib.slx`
- Impact: Cannot work with earlier MATLAB versions; hard to diff/merge model changes in git; binary merge conflicts unresolvable
- Fix approach: Document version compatibility matrix; establish SLX diff/merge strategy; consider autogenerated model architecture from description

**Inertia Parameter Unmeasured:**
- Issue: Joint 1 rotational inertia `Iz_1 = 3e-5 kgm²` is never measured, only design-estimated
  - Comment in `scripts/stepper/RRpendulum_FSFB_sensitivity_analysis.m:13`: "design value (HAS NOT BEEN MEASURED)"
  - Sensitivity analysis `scripts/stepper/RRpendulum_FSFB_sensitivity_analysis.m` shows stability margin decreases sharply if real `Iz_1 > 1.5x` design value
- Files: `scripts/BLDC/RRpendulum_Parameters_num_BLDC.m:9`, `scripts/stepper/RRpendulum_Parameters_num.m` (missing entirely), all control design files
- Impact: Observer-based state feedback controller gains (`Kd`) may be under-damped or unstable if hardware inertia differs; swing-up energy controller torque commands depend on pose-dependent inertia calculation
- Fix approach: Measure `Iz_1` experimentally (oscillation test or CAD); re-tune controller gains with measured value; add bounds check in initialization

## Security Considerations

**Firmware Path Hard-Coded in Script:**
- Risk: `scripts/BLDC/RRpendulum_BLDC_drive_params.m:17` hard-codes local path `'C:/Users/u0130154/STM32_vscode_projects/invpend_BLDC/Inc'`
  - Script reads C header files to parse motor/drive parameters
  - Path contains username and is Windows-specific
  - If moved or shared, script silently fails to find firmware headers
- Files: `scripts/BLDC/RRpendulum_BLDC_drive_params.m:17`
- Current mitigation: Script has comment warning to "change into a path on your PC"
- Recommendations: 
  - Parameterize firmware path as input or environment variable
  - Add error checking if header file not found
  - Document header file path setup in CLAUDE.md

**No Input Validation in Dynamics Functions:**
- Risk: Auto-generated dynamics function `resources/functions/RRpendulum_dynamics_ct.m` accepts arbitrary input without bounds checking
  - Called by UKF observer `ukf_observer_step.m` which has NaN/Inf firewall (lines 33–36) but base dynamics has none
  - If observer state estimate becomes NaN, dynamics evaluation will silently propagate it
- Files: `resources/functions/RRpendulum_dynamics_ct.m`
- Current mitigation: UKF wrapper checks for non-finite estimates and resets to [0;0;0;0]
- Recommendations: Add saturation/clamping to dynamics function or document that all callers must validate inputs

## Performance Bottlenecks

**High-Dimensional Symbolic Derivation (slow regeneration):**
- Problem: `scripts/RRpendulum_forkin_dyn_noimage.m` derives 2-DOF kinematics and dynamics symbolically from scratch every run
  - Uses Symbolic Math Toolbox to compute Jacobians, kinetic energy, potential energy, and Lagrange equations
  - Generates ~317 lines of symbolic derivation and substitution
  - Saves derived EOM to `data/RRpendulum_EOM.mat` but must re-run to refresh
- Files: `scripts/RRpendulum_forkin_dyn_noimage.m`
- Cause: No caching; symbolic derivation is O(n⁴) complexity for Lagrangian approach
- Improvement path: 
  - Cache symbolic EOM in version-controlled `.mat` file with hash verification
  - Add conditional re-derivation only if symbolic parameters change
  - Document derivation step as optional (one-time) setup rather than per-session

**UKF Observer Integration with Dynamics Overhead:**
- Problem: `resources/functions/ukf_observer_step.m` integrates nonlinear dynamics 9 times per sample (one for each sigma point) using RK4
  - At 1 kHz sample rate, this is 9000 RK4 steps/second
  - Each RK4 step calls `RRpendulum_dynamics_ct` 4 times
  - No vectorization; loop-based (lines 61–64)
- Files: `resources/functions/ukf_observer_step.m:61-64`, called from Simulink block
- Cause: Unscented Kalman Filter intrinsic complexity; autogenerated dynamics function not SIMD-optimized
- Improvement path: 
  - Benchmark observer CPU time on target hardware (ARM MCU)
  - Consider simpler observer (e.g., alpha-beta filter) for real-time embedded use
  - Vectorize sigma point propagation if switching to CPU-based observer

## Fragile Areas

**UKF Cholesky Decomposition Fallback Chain:**
- Files: `resources/functions/ukf_observer_step.m:40-50`
- Why fragile: Covariance matrix `P` can become non-positive-definite due to accumulated numerical errors
  - Cholesky decomposition fails → adds diagonal loading 1e-8 and retries
  - If that fails → uses scaled identity `1e-6 * eye(4)` as emergency repair
  - No logging of when fallback occurs; silent recovery masks underlying filter divergence
- Safe modification: Add warning/diagnostic flag when fallback is triggered; monitor in closed-loop tests
- Test coverage: No unit tests for UKF observer; only used in full-system Simulink simulation

**Cross-Script Dependencies via MAT Files:**
- Files: `scripts/BLDC/RRpendulum_UKF_design.m` loads `data/RRpendulum_EOM.mat`, `scripts/BLDC/RRpendulum_swingup_controldesign.m` loads `data/FSFB_torque_design.mat`
- Why fragile: MAT files are binary; if corrupted or deleted, dependent scripts fail silently until runtime
  - No hash verification of loaded data
  - Dependency chain is implicit (buried in `load()` calls)
  - Running scripts out of order produces stale designs
- Safe modification: Add DAG (directed acyclic graph) comments documenting load order
- Test coverage: Manual execution order required; no automated dependency verification

**Discretization Margin at 1 kHz:**
- Files: `scripts/analyze_ZOH_performance.m`, `scripts/stepper/RRpendulum_FSFB_sensitivity_analysis.m`
- Why fragile: ZOH discretization at 1 kHz with recommended `R=5000` (from `analyze_ZOH_performance.m`) produces closed-loop poles near Nyquist frequency
  - Fastest CL pole: ~390 rad/s = 62 Hz (expected from analysis)
  - Nyquist frequency: 500 Hz
  - Margin: only 8x
  - If loop execution is delayed or jittered, stability margin erodes
- Safe modification: Lower `R` further (increase control effort cost) to push CL poles below 100 rad/s; add real-time latency model to sensitivity analysis
- Test coverage: Stability robustness only analyzed offline; no real-time jitter injection test

**Observer Bandwidth Limited by Encoder Quantization:**
- Files: `scripts/BLDC/RRpendulum_decoupledLuenberger.m:64-70` (obsolete but documents principle)
- Why fragile: Joint 2 encoder has 2400 counts/turn (stepper has 8192)
  - Quantization frequency: ω_quant ≈ 1.6 krad/s
  - 1/5 rule suggests observer bandwidth ≤ 320 rad/s
  - But `scripts/stepper/RRpendulum_FSFB_sensitivity_analysis.m` tests observer poles as fast as [-400, -350, -300, -250] rad/s
  - Result: noise amplification and chatter on torque commands
- Safe modification: Run full UKF noise sensitivity test with real encoder data; compare predicted vs. actual velocity noise
- Test coverage: Theoretical analysis only; no hardware-in-loop validation of observer performance

## Scaling Limits

**Multiple Model Variants Without Clear Routing:**
- Current capacity: 5 control strategies × 2 hardware platforms (stepper/BLDC) = 10 model files
  - `models/stepper/RRpendulum_digtwin.slx` (kinematic control)
  - `models/stepper/RRpendulum_digtwin_FSFB.slx` (state feedback torque)
  - `models/stepper/RRpendulum_digtwin_swingup*.slx` (swing-up energy control)
  - `models/BLDC/RRpendulum_digtwin_BLDC.slx`, `RRpendulum_digtwin_FSFB_BLDC.slx`, etc.
- Limit: At 15+ models, versioning and cross-platform testing become manual and error-prone
- Scaling path: 
  - Consolidate to single model with Simulink Variant blocks (one per control strategy)
  - Use model reference for shared hardware interface library (`resources/lib/digtwin_labo_lib.slx`)
  - Automate code generation for each variant

**Single Data Directory for All Designs:**
- Current capacity: `data/` stores 6 MAT files (EOM, parameters, FSFB design, UKF design, swingup design, video)
- Limit: No subdirectories or versioning; all scripts load from same location with no namespace separation
- Scaling path: Organize by design type (`data/designs/FSFB_torque/`, `data/designs/UKF/`, etc.) and add timestamp/git-hash versioning

## Dependencies at Risk

**Symbolic Math Toolbox Critical for Offline Design:**
- Risk: Design chain depends on Symbolic Math Toolbox (symbolic derivation, linearization, Jacobians)
  - `scripts/RRpendulum_forkin_dyn_noimage.m` uses `sym`, `jacobian`, `simplify`
  - `scripts/BLDC/RRpendulum_UKF_design.m` uses `matlabFunction` to generate `RRpendulum_dynamics_ct.m`
  - If Symbolic Math Toolbox is removed or license expires, design workflow breaks
- Impact: No alternative symbolic derivation path; controller gains cannot be re-tuned
- Migration plan: Pre-generate and commit `RRpendulum_dynamics_ct.m` and linearized matrices; document that symbolic derivation is optional offline step

**Hard-coded MATLAB Paths in Scripts:**
- Risk: Scripts call `openProject('C:/Users/u0130154/MATLAB/projects/digtwin_labo/digtwin_labo.prj')` with absolute Windows path
  - Works only on one user account on one machine
  - Fails on Linux or macOS
- Files: `scripts/BLDC/RRpendulum_Parameters_num_BLDC.m:56`, `scripts/BLDC/RRpendulum_UKF_design.m:16`, `scripts/RRpendulum_UKF_design.m` (multiple places)
- Impact: Cannot run scripts on different machines or users without editing
- Migration plan: Use relative path `fullfile(fileparts(mfilename('fullpath')), '..', 'digtwin_labo.prj')` or environment variable

## Missing Critical Features

**No Real-Time Validation Framework:**
- Problem: All control designs are validated offline in MATLAB/Simulink; no formal real-time latency, jitter, or quantization budget
  - Assume 1 kHz sample time but don't measure actual loop execution time
  - UKF observer is validated symbolically but not on target ARM hardware
  - No hardware-in-loop test harness
- Blocks: Cannot verify that control objectives are met in production
- Solution: Build real-time test model with instrumentation for latency, compute load, state estimation error

**No Continuous Integration or Automated Testing:**
- Problem: Scripts run manually; no regression testing when parameters or algorithms change
  - No test suite for symbolic derivation consistency
  - No automated checks that controller gains stabilize the system
  - No verification that code generation produces runnable embedded C code
- Blocks: Risk of silent failures if someone modifies a core script
- Solution: Write parametric MATLAB unit tests; set up GitHub Actions to run key scripts on every commit; save golden reference designs and compare

**No Operational/Embedded Documentation:**
- Problem: Design decisions and safety constraints are scattered across scripts and comments
  - CLAUDE.md provides overview but lacks implementation details
  - No "operator's manual" explaining how to swap controllers or retune gains
  - No embedded C code comments linking generated code back to MATLAB design
- Blocks: Difficult for embedded firmware developers to understand what the generated C code is supposed to do
- Solution: Auto-generate annotated C with Embedded Coder comments; write DESIGN.md explaining each controller variant

## Test Coverage Gaps

**Symbolic Derivation Not Verified:**
- What's not tested: Forward kinematics, Lagrangian formulation, Euler-Lagrange EOM derivation
  - `scripts/RRpendulum_forkin_dyn_noimage.m` computes symbolic equations but only outputs them to workspace
  - No unit tests that verify equations match expected pendulum dynamics
  - No comparison against published pendulum models or CAD properties
- Files: `scripts/RRpendulum_forkin_dyn_noimage.m`, `resources/functions/derive_EOM.m`
- Risk: Subtle errors in kinematics (e.g., DH parameter sign, reference frame orientation) would propagate to controller design and hardware
- Priority: **High** — wrong symbolic model is the root cause of all downstream errors

**Linearization Validity Not Checked:**
- What's not tested: Linearization around `(q2=π, v2=0, v1=0)` is only valid for small perturbations
  - No analysis of domain of validity (how far from equilibrium before linearization breaks down)
  - No Lyapunov stability proof that linear controller stabilizes nonlinear system
  - Test files: none; only offline analysis
- Files: `scripts/BLDC/RRpendulum_FSFB_controldesign_torque.m:45-58`
- Risk: Controller may fail if pendulum deviates >0.5 rad from vertical; no early warning
- Priority: **High** — swing-up must hand off to linear controller, and handoff region is not validated

**UKF Observer Numerical Stability Not Tested:**
- What's not tested: Cholesky decomposition repair (lines 42–50 of `ukf_observer_step.m`) is empirically validated but no bounds proof
  - No test of observer divergence under sensor dropout
  - No test of observer under extreme initial state (e.g., pendulum swinging at ±100 rad/s)
  - No batch data set with real sensor noise for filter tuning
- Files: `resources/functions/ukf_observer_step.m`, `scripts/BLDC/RRpendulum_UKF_design.m`
- Risk: Observer becomes unstable and resets to [0;0;0;0] silently; control loop loses state estimate
- Priority: **Medium** — fallback exists but is not logged; difficult to diagnose in hardware

**Hardware Interface Library Never Tested in Isolation:**
- What's not tested: `resources/lib/digtwin_labo_lib.slx` communicates with MCU firmware via SPI at 2 MHz
  - No unit test that verifies SPI packet format, CRC, timeout handling
  - No test of MCU communication latency or packet loss recovery
  - Code generation (Embedded Coder → C) from this library is not validated standalone
- Files: `resources/lib/digtwin_labo_lib.slx`
- Risk: Communication corruption (bit flip, packet loss) could cause erratic behavior; no error detection
- Priority: **High** — interface is critical path to hardware; any bug blocks deployment
- Current doc: CLAUDE.md states library "must stay synchronized with MCU firmware" but no verification method given

**No Parameter Sensitivity Automated Regression:**
- What's not tested: `scripts/stepper/RRpendulum_FSFB_sensitivity_analysis.m` is a one-off analysis
  - No automated regression when physical parameters (mass, length, damping) are updated
  - No test that verifies redesigned controller still stabilizes under parameter mismatch
- Files: `scripts/stepper/RRpendulum_FSFB_sensitivity_analysis.m`
- Risk: If someone updates mass estimate without re-running sensitivity analysis, stability margin is lost silently
- Priority: **Medium** — sensitivity analysis exists but must be run manually after parameter changes
