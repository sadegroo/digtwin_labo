# Coding Conventions

**Analysis Date:** 2026-04-02

## Naming Patterns

**Files:**
- Script format: PascalCase with underscores, topic-descriptive names
  - Example: `RRpendulum_Parameters_num.m` (parameter definitions with numeric suffix)
  - Example: `RRpendulum_FSFB_controldesign_accel.m` (control design, specific approach)
  - Pattern: `[RobotName]_[FunctionArea]_[Specifics].m`
- Auto-generated functions: Similar pattern with generation metadata
  - Example: `RRpendulum_dynamics_ct.m` (continuous-time dynamics, auto-generated by matlabFunction)
  - Example: `RRpendulum_totalIz1.m` (inertia calculation, auto-generated)
- Variant management: Live Script (`.mlx`) + plain `.m` pairs for programmatic use
  - Example: `RRpendulum_forkin_dyn.mlx` (Live Script with visualization) + `RRpendulum_forkin_dyn_noimage.m` (headless version)

**Variables:**
- **Symbolic variables:** Lowercase with underscores, explicit `(t)` for time-dependent
  - Example: `q_1(t)`, `q_2(t)`, `v_1(t)`, `v_2(t)` in symbolic derivations
  - Pattern: `q_i` = joint angle (rad), `v_i` = angular velocity (rad/s)
- **Numeric parameters:** Lowercase ending with `_num` suffix (single source of truth convention)
  - Example: `m_num = 0.237` (mass in kg)
  - Example: `l_num = 0.235` (length in m)
  - Example: `Iz_1_num = 3e-5` (inertia kg·m²)
  - Pattern: Physical quantity name + `_num` → ensures distinction from symbolic equivalents
- **States:** Compact vector notation
  - Example: `z = [q1; v1; q2; v2]` or `x = [q(1); v(1); q(2); v(2)]`
  - Pattern: Consistent state ordering across all scripts (position, velocity pairs)
- **Matrices:** Uppercase for system matrices, lowercase for intermediate results
  - Example: `A_lin`, `B_lin`, `A_sym`, `B_sym` (linearized vs symbolic)
  - Example: `M_mat`, `C_mat`, `G_vec` (inertia, Coriolis, gravity matrices)
  - Example: `K` (LQR gain), `L` (observer gain), `Kd` (discrete LQR gain)

**Functions:**
- Camel case with descriptive names, parameters in meaningful order
  - Example: `RRpendulum_dynamics_ct(in1, tau_1)` — state input, control input
  - Example: `swingup_energy_controller(q1, v1, q2, v2, k_e, alpha_max, E0, m, l, g, r, Iz_1)` — joint states, gains, physical params
  - Example: `ukf_observer_step(x_hat, P, u, y, ...)` — standard UKF argument order
  - Example: `rotationMatrixToZYXEuler(R)` — single descriptive verb
- Helper functions: Prefixed with lowercase, placed as nested functions
  - Example: `rk4_step(x, u, dt)` inside `ukf_observer_step` — Runge-Kutta integrator
- Constraint: Parameter order convention: states → inputs → gains/tuning → physical parameters

**Types:**
- All variables are numeric (double precision) unless explicitly symbolic
- Symbolic Math Toolbox creates symbolic variables via `syms` declarations at script top
- No explicit type annotations (MATLAB infers types automatically)

- **Spacing:**

  - No blank lines within compact loops or matrix definitions
- **Matrix notation:** Multi-line matrices align on `[` brackets
  - Example (from `DH_full.m`): DH transformation matrices written explicitly with coordinate columns aligned

**Linting:**
- No explicit linter configuration detected (`.eslintrc`, `.mlintrc` not present)
- Code adheres to MATLAB R2025b standards (no deprecated syntax)

## Import Organization

**Script execution order (critical dependency):**
1. Symbolic derivation: `run RRpendulum_forkin_dyn_noimage.m`
   - Defines symbolic variables: `q`, `v`, `q_dot`, `q_ddot`
   - Defines Lagrangian `L`, equations of motion `EOM`
   - Outputs: symbolic dynamics `eqn_vdot1`, `eqn_vdot2`
2. Parameter definition: `run RRpendulum_Parameters_num.m`

3. Control/Observer design: e.g., `run RRpendulum_FSFB_controldesign_accel.m`

## Error Handling

**Input validation:**
- Explicit error checks in critical functions
  - `rotationMatrixToZYXEuler.m`: `if ~isequal(size(R), [3, 3]) error('Input matrix must be 3x3.');`
  - Pattern: Check matrix dimensions before mathematical operations
- Auto-generated functions (e.g., `RRpendulum_dynamics_ct.m`) assume valid inputs (no checks)

**NaN/Inf firewall:**
- `ukf_observer_step.m` includes explicit robustness check (line 33):
  ```matlab
  if any(~isfinite(x_hat(:))) || any(~isfinite(P(:)))
      x_hat_new = [0; 0; 0; 0];   % pendulum hanging down (safe)
      P_new = Q * 1e3;
      return;
  end
  ```
  - Prevents propagation of invalid estimates through Unscented Kalman Filter
  - Returns safe default state (pendulum hanging) rather than propagating garbage

**Numerical robustness:**
- Covariance matrix symmetry enforced (line 41, 75, 97 in `ukf_observer_step.m`):
  ```matlab
  P_scaled = 0.5 * (P_scaled + P_scaled');  % enforce symmetry
  ```
- Cholesky decomposition with fallback repair (lines 42-50):
  - Attempt Cholesky with `'lower'` flag
  - If singular: add 1e-8 diagonal loading and retry
  - Last resort: use scaled identity matrix for regularization
- Covariance bounds enforce physical limits (lines 100-108):
  - Floor: 1e-10 (near-zero uncertainty not believed)
  - Ceiling: `[40; 1e4; 40; 1e4]` (per-state physical bounds)

**Parameter saturation:**
- Control saturation implemented in `swingup_energy_controller.m` (lines 40-43):
  ```matlab
  if alpha1_des > alpha_max
      alpha1_des = alpha_max;
  elseif alpha1_des < -alpha_max
      alpha1_des = -alpha_max;
  end
  ```

## Comments

**When to Comment:**
- **Above function definitions:** Full docstring in help format (MATLAB standard)
  - Example from `swingup_energy_controller.m`:
    ```matlab
    %   Implements the Åström-Furuta (Automatica, 2000) energy control law
    %   tau1 = Iz1_total(q1, q2) · alpha1
    %   Inputs: [list of parameters with units]
    %   Output: [description with units]
    ```
- **At complex algorithmic steps:** Explain the "why" not the "what"
  - Example (line 93 in `ukf_observer_step.m`): `% Kalman gain` (concise, code speaks for itself)
  - Example (line 95-96): `% State and covariance update (Joseph form — guaranteed PSD)`
- **For non-obvious numerical values:** Explain units and rationale
  - Example (line 71 in `ukf_observer_step.m`): `v_max = 100;  % rad/s — generous physical bound`
  - Example (line 100-101): `P_floor = 1e-10;` followed by `P_ceil = [40; 1e4; ...]  % [pos; vel; pos; vel]`

**Live Script Rich Text Format:**
- Plain `.m` files use `%[text]` blocks for markdown-like documentation
  - Supports **bold**, `code`, LaTeX `$...$` inline math
  - Line escaping: `\_` for underscore, `\\` for backslash in math mode
  - Pattern: One-line summaries followed by detailed explanation blocks
  - Example: `%[text] # FSFB Torque Controller — Sensitivity Analysis`
- Symbolic output tagged with `%[output:hex_id]` for linked cell outputs (Live Editor feature)
  - Not human-readable (auto-generated by MATLAB)

**JSDoc/TSDoc:** Not applicable (MATLAB-specific help format used instead)

**Symbolic notation in comments:**
- Uses LaTeX for mathematical expressions: `$\\dot{x} = Ax + Bu$`
- Array notation: `x\_i = \[q\_i;; \\dot{q}\_i\]` (note: doubled semicolon for readable column vector)
- Example (line 32 in `RRpendulum_decoupledLuenberger.m`):
  ```matlab

  %[text] with $x\_i = \[q\_i;; \\dot{q}\_i\]$
  ```

## Function Design

**Size:** 10-50 lines typical (auto-generated functions can be 30 lines of compact math)

**Parameters:**
- Ordered consistently: states → inputs → gains/tuning → physical constants
- Example order from `swingup_energy_controller`:
  1. Joint states: `q1, v1, q2, v2`
  2. Control gain: `k_e`
  3. Saturation limit: `alpha_max`
  4. Setpoint: `E0`
  5. Physical parameters: `m, l, g, r, Iz_1`
- No default parameter values (all explicitly passed)

**Return values:**
- **Single output:** Returned directly without explicit `output =` syntax
  - Example: `function tau1 = swingup_energy_controller(...)`
  - Example: `function Iz1tot = RRpendulum_totalIz1(...)`
- **Multiple outputs:** Comma-separated in return signature
  - Example (from `DH_full.m`): `function [ Tfull, Tparts, Tcumul ] = DH_full( DH_table )`
  - Example (from `derive_EOM.m`): `function [EOM, lambda_eqs] = derive_EOM(L, q, Q, constraints)`
- Auto-generated functions use compact intermediate variable names
  - Example (from `RRpendulum_dynamics_ct.m`): `t2 = cos(q_2); t3 = sin(q_2); ... xdot = [mt1; mt2];`

## Module Design

**Exports:**
- All functions in `resources/functions/` are public (no underscore prefix convention)
- Scripts generate workspace variables (no module concept; everything is global after `run`)

**Barrel files:** Not used (MATLAB project path mechanism instead)

**Organization principle:** Symbolic generation → numeric substitution → controller design
- Symbolic math kept separate (`derive_EOM`, `DH_full`)
- Auto-generated numeric functions stored in `resources/functions/`
- Control scripts in `scripts/` call both, then design controllers

**Naming scoping:**
- No nested namespaces (MATLAB R2025b doesn't support package nesting used here)
- Functions organized by purpose: dynamics, kinematics, control, estimation
- Cross-script variables rely on consistent naming convention (`*_num` for numeric parameters)
